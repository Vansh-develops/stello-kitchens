# Petpooja Teardown & Buildable Engineering Specification for a Full Clone

## TL;DR
- Petpooja is India's largest restaurant-management SaaS. Its September 2025 Series C "valued Petpooja at around Rs 910 crore ($103 million), a 3.5X jump from its previous round" (Entrackr), with Dharana Capital contributing ₹82 cr (18.62% fully-diluted stake), Ashish Gupta (Helion co-founder) ₹1 cr, and Urban Company founders Abhiraj Singh Bhal, Varun Khaitan and Raghav Chandra also participating. It is an offline-first, cloud-synced POS + KOT + KDS + inventory/recipe + menu + CRM/loyalty + reporting + multi-outlet platform wrapped around ~150+ third-party integrations and a marketplace of add-on apps (Captain App, KDS, Scan & Order, Kiosk, Token, Wireless Calling, Digital Menu Board). Per Petpooja's Series C release, it "processes over 7 million orders daily and around 25% of online order volumes on Zomato and Swiggy" across 100,000+ restaurants in 140+ cities.
- You can build ~80% of it directly (POS, KDS, inventory, menu, CRM, reporting, multi-tenant control, offline sync, ESC/POS printing). Three categories must be rented or integrated rather than built: aggregator order feeds (Zomato needs approved-POS-vendor onboarding with a 99.999% uptime SLA; Swiggy is partner-gated — or use middleware like UrbanPiper/Dyno), payments (Razorpay/Paytm/PhonePe), and GST e-invoicing/accounting (integrate Tally/IRP via a GSP rather than build GSTN plumbing).
- Recommended stack: an **offline-first edge POS** (Electron on Windows + Kotlin/Android, each with a local SQLite/embedded store and a local print/KDS LAN server) syncing to a **NestJS + PostgreSQL + Prisma + Redis + BullMQ** multi-tenant backend, with a dedicated event-driven **aggregator connector service**, WebSocket layer for KDS, and Next.js dashboards. This maps directly onto your existing Node/NestJS/PostgreSQL/Prisma/BullMQ/Redis/Next.js competency; the only genuinely new muscle is the local-first edge client and the printer/hardware layer.

## Key Findings

### What Petpooja actually is
Confirmed via YourStory (30 Sep 2025): "Founded in 2011 by Parthiv and Apurv Patel, Petpooja serves over 100,000 restaurants in India, the Middle East, the US, and Africa, across 140 cities." It began as a B2B corporate meal-delivery business (serving 200+ corporates / 300+ restaurants) before launching its configurable billing platform in November 2013; the legal entity is Prayosha Food Services Pvt. Ltd. (Ahmedabad). Petpooja is a cloud-based but offline-capable restaurant operating system. It serves QSRs, cafes, bakeries, cloud kitchens, fine-dining, food courts, dessert/ice-cream shops, pizzerias, bars/breweries and multi-outlet chains across India, UAE, Middle East, Canada and South Africa. It has expanded beyond POS into a broader SME suite (payroll + biometric hardware, a retail/B2B "Petpooja Invoice" GST product, AI invoice processing, task management). Per Entrackr, "the firm saw its revenue surge 43% to Rs 76 crore in FY24" while "losses also narrowed to Rs 13.4 crore"; Petpooja has raised ~₹185 crore across rounds total.

The product's defining architectural characteristics for a cloner to replicate:
1. **Offline-first billing** — the POS keeps working with no internet; KOTs reach kitchen printers/KDS over the local network; cloud is only needed for sync and pulling aggregator orders.
2. **Single-screen consolidation** — dine-in, takeaway, delivery and all aggregator channels on one billing screen.
3. **Multi-terminal, master-station sync** — multiple billing counters sync to one master station; each cooking station gets its own printer/KDS.
4. **Everything is multi-outlet/multi-brand aware** from the data model up.

### How aggregator integration actually works (the critical technical dependency)
Petpooja has been a Zomato POS partner since 2017. The Zomato POS Integration API model, which your clone must replicate to be a direct partner:

- **Onboarding path (approved POS vendors only):** submit the "getting started" form → Zomato POC responds → sign NDA → fill Vendor Onboarding Form (base URLs, webhook endpoints, header/auth-token configs, POS api key) → Zomato POC **creates a POS ID** and configures webhook URLs → whitelist domains, IPs and test-store email IDs → build against mock server + sandbox → certification → **controlled pilot with two restaurants** → gradual rollout to all outlets.
- **Prerequisites (hard gates):** 100% feature parity on the critical-feature list; a 24×7 Slack/WhatsApp on-call channel with tech/product/ops PICs and **<10-minute turnaround**; **system uptime > 99.999%**.
- **Three API surfaces + webhooks:** Menu Management (add/upsert menu, get menu, item stock toggle via `menu/item/stock`, plus menu-processing and menu-moderation callbacks), Order Management (confirm/accept, mark ready-for-pickup, mark picked up, assign delivery partner, mark delivered, request masked customer/rider contact, confirm/reject cancellations, complaints), Outlet Management (delivery status get/update at `/online-ordering/v1/restaurant_delivery_status/get|update`, delivery-time & surge, Zomato vs self-delivery timings, logistics status). Base host `https://api.zomato.com`, paths under `/online-ordering/v1/...`, JSON over REST.
- **Order lifecycle & statuses:** Zomato calls the POS **Order Relay Webhook** with a new order → POS confirms → prepares → marks ready → for Zomato logistics, Zomato assigns rider (`rider-assigned`) then pickup (`pickedup`); for self-logistics the POS assigns/dispatches/confirms delivery. Rejections use `reject`; timeouts use `timedout`. **Auto-reject:** if the restaurant doesn't act within a defined inaction window (e.g., 5 minutes), Zomato fetches order status via the Fetch Order Status webhook and, if still unconfirmed/rejected, automatically rejects. Key gotcha: **triggering the add-menu API toggles the outlet off search internally** until processing completes, and the menu request is a full snapshot (only entities sent are retained).
- **Swiggy** is partner-gated (developers.swiggy.com; API key + secret, token-based auth) with no public endpoint docs; parallels Zomato (menu sync/bulk upload, item toggle, live order fetch with New/Accepted statuses, masked customer details). Swiggy built its own 3rd-party Order Management API in 2017 and its Cloud Menu sync cut menu-change propagation from ~2 days to ~2 seconds.
- **Middleware shortcut:** UrbanPiper (Hub), Dyno APIs, and Deliverect provide a single integration that fans out to Swiggy/Zomato/Uber Eats/Magicpin/etc., handling order relay, menu sync, stock sync and reconciliation. This is the fastest path if you don't want to pass each aggregator's onboarding gate yourself.

### How ONDC seller-side works
ONDC is a Beckn-protocol network, not an app. To sell food you become a **Seller Network Participant (SNP)** or connect through one. You implement the protocol APIs (`/search`, `/select`, `/init`, `/confirm`, `/status`, `/track`, `/cancel`, `/update`, `/rating`, `/support`) and their callbacks (`/on_search`, `/on_select`, `/on_init`, `/on_confirm`, etc.). Onboarding: register/whitelist a subscriber ID in the ONDC Registry (staging → pre-prod → production), sign requests digitally (ed25519 signatures), and pass end-to-end testing with ONDC reference buyer apps. Food & Beverage sits under retail domain code **RET11**. Practical path (Petpooja's own): partner with an existing SNP (GrowthFalcons) rather than becoming a full network participant on day one.

### GST / e-invoicing / compliance a POS must handle
- **GST invoice** must carry seller & buyer GSTIN, HSN/SAC codes, per-slab CGST/SGST/IGST breakup, place of supply, taxable value. Restaurants typically bill 5% GST (no ITC) for restaurant service; packaged goods carry their own HSN slabs.
- **E-invoicing (IRN/QR):** mandatory for businesses above ₹5 crore aggregate turnover (PAN-level). Invoice JSON (schema GST INV-01) is uploaded to the **IRP**, which returns an **IRN + digitally-signed QR**. Per GSTN advisory dated 5 Nov 2024: "from 1st April 2025, taxpayers with an AATO of 10 crores and above would not be allowed to report e-Invoices older than 30 days from the date of reporting on IRP portals" — lowered from the prior ₹100 crore threshold (in force since 1 Nov 2023); applies to invoices, credit notes and debit notes. From 1 June 2025 invoice numbers are treated case-insensitively (uppercased). 2FA mandatory for IRP/e-way-bill portals. E-invoices cannot be modified — corrections go via credit/debit notes and GSTR-1 amendments.
- **GSTR-1/3B:** e-invoice data auto-populates GSTR-1; Table 12 HSN now uses a dropdown (no manual entry) split B2B/B2C. GSTR-9C self-certified reconciliation for turnover > ₹5 crore.
- **E-way bills:** for goods movement (e.g., central-kitchen transfers); generated after IRN; recent advisories require Ship-to GSTIN on inter-branch transfers.
- **Design implication:** a POS should **store** GSTIN/HSN/tax-slab/place-of-supply and the returned IRN/QR fields, and delegate actual IRN generation and GSTR filing to an accounting integration or a **GSP/ASP** (via API) rather than building GSTN plumbing directly. This is how Petpooja does it (Tally sync; Petpooja Invoice has native e-invoice/e-way-bill generation through a GSP).

### Offline-first, printing and real-time architecture
- **Offline-first:** each POS/Captain device keeps a **local database** as the primary source of truth, applies **optimistic local writes**, and syncs **deltas** to the cloud when connectivity returns using **last-write-wins with version vectors / Lamport clocks + tombstones** for deletes; bill-number sequences are maintained both offline and online. A local **master station** on the LAN coordinates multiple terminals and drives printers/KDS even with the WAN down.
- **Printing:** ESC/POS command set over USB/LAN/Bluetooth for 58mm/80mm receipt & KOT printers; StarPRNT for Star devices; ZPL/CPCL for label printers; Android print-service intents on tablets; a desktop print service (Electron native or QZ-Tray-style) for silent printing. Template engine renders bill/KOT/label layouts.
- **Real-time KDS:** event-driven; a local WebSocket/LAN broadcast delivers punched orders to station screens within ~2s; cloud WebSocket layer pushes aggregator orders and cross-device updates.

## Details — The Buildable Engineering Specification (for Claude Code CLI)

### Recommended technology stack (and where your existing stack fits)
The workload has five hard requirements: offline-first edge billing, real-time KDS, multi-tenant multi-outlet, high-volume order ingest, and hardware/printer + multi-channel sync. Recommended:

- **Cloud backend:** **NestJS (Node/TypeScript)** modular monolith with clear module boundaries → extract the aggregator connector and sync service as separate deployables early. **PostgreSQL 15+** (with **row-level multi-tenancy** via `tenant_id`/`outlet_id` and RLS), **Prisma** ORM, **Redis** (cache + pub/sub), **BullMQ** (jobs: menu push, reconciliation, report generation, retries), **Kafka or Redis Streams** for the high-volume order-event bus if you scale past a single connector node. **This is exactly your Node/NestJS/PostgreSQL/Prisma/BullMQ/Redis/Next.js competency** — keep it for the entire cloud tier.
- **Real-time layer:** WebSockets via **NestJS Gateway (Socket.IO)** for KDS/owner-dashboard cloud fan-out; **MQTT or a lightweight local WS server** on the in-outlet master device for LAN delivery when offline.
- **Edge/POS client (the genuinely new part):** **Electron + React + TypeScript** on Windows (dominant Indian POS OS) and **Kotlin/Android** (or React Native/Expo) for Captain/KDS/kiosk. Local store: **SQLite** (better-sqlite3 on Electron; Room on Android). Consider a sync engine (ElectricSQL/PowerSync/Ditto-style CRDT sync) or a hand-rolled delta-sync queue. Each outlet runs a **local master service** (bundled with the primary POS or a small always-on box) that owns the LAN, printers and KDS broadcast.
- **Printer/hardware layer:** a device-side print service speaking ESC/POS/StarPRNT/ZPL over USB/LAN/BT; abstract behind a `PrinterDriver` interface.
- **Web dashboards & storefront:** **Next.js** (owner/admin dashboard, menu/inventory management, reporting, and the first-party ordering website + Scan & Order PWA).
- **Where Node/NestJS falls short — be honest:** Node is fine for the cloud API and connector, but the **offline edge client is not a Node-web problem** — it needs a real embedded DB and native print/peripheral access, hence Electron/Kotlin. Extremely high-fan-out real-time and heavy reconciliation number-crunching are better as isolated workers; don't try to serve KDS sockets and aggregator webhooks from the same process that renders dashboards. For the 99.999% aggregator-facing uptime SLA, the connector must be its own horizontally-scaled, multi-AZ service with its own on-call.
- **Hosting/infra:** containerized (Docker) on Kubernetes or ECS, multi-AZ; managed Postgres (RDS/Cloud SQL) with read replicas; object storage for reports/exports; per-outlet edge devices as the offline tier.

### Modular architecture breakdown
Multi-tenancy model: **Tenant (brand group) → Brand → Outlet → Terminal/Device**, with `tenant_id` on every row, outlet-scoped RBAC, and per-outlet price/availability overrides. Services/modules:
1. **Identity & Access** (tenants, users, roles, permissions, device registration, audit logs).
2. **Catalog/Menu** (categories, items, variations, add-ons, combos, modifiers, per-channel menus, pricing, timings, out-of-stock).
3. **Ordering/POS core** (orders, order-types, KOTs, tables/areas, split/merge, running orders, discounts/offers/happy-hours).
4. **KDS/real-time** (station routing, bump, prep timers) — WS.
5. **Inventory** (raw materials, recipes multi-stage, auto-deduction, POs, vendors, transfers, wastage, variance, central kitchen/indenting, forecasting).
6. **CRM/Loyalty/Marketing** (customers, wallet/points, coupons, segmentation, campaigns, feedback).
7. **Payments** (gateway abstraction, split payments, dynamic UPI QR, reconciliation, cash drawer/day open-close/expenses).
8. **Aggregator Connector** (separate service: Zomato/Swiggy/ONDC adapters, order ingest, menu push, stock sync, status callbacks, webhook receiver, per-adapter rate-limit/retry).
9. **Reporting/Analytics** (materialized reports, dynamic report builder, dashboards).
10. **Tax/Compliance** (GST config, HSN, IRN/QR storage, e-way bill, Tally/GSP export).
11. **Sync service** (delta sync, conflict resolution, device state, bill-number authority).
12. **Notifications** (SMS/WhatsApp/email providers).
13. **Hardware/Print** (device-side driver + template engine).

### Build vs. rent vs. integrate — explicit flags
- **Build directly:** POS/billing/KOT, KDS, menu, inventory/recipes/central kitchen, CRM/loyalty, reporting, multi-outlet/RBAC, offline sync, ESC/POS printing, first-party ordering/Scan & Order/kiosk/token, cash management.
- **Rent / third-party connectivity required:** aggregator order feeds — either pass **Zomato's approved-POS-vendor onboarding** (NDA, POS ID, 99.999% uptime, <10-min on-call) and **Swiggy's partner gate**, or ride **UrbanPiper/Dyno/Deliverect** middleware. ONDC via an existing **SNP (GrowthFalcons-style)** before full network participation. SMS/WhatsApp via MSG91/Gupshup.
- **Integrate rather than build:** payments (Razorpay/Paytm/PhonePe — never handle card PANs yourself), GST e-invoicing/IRN & e-way bills (GSP/ASP), accounting (Tally/Zoho), logistics (Shadowfax/Porter/Dunzo/Pidge/Tookan).

## Recommendations
1. **Sequence:** ship Phases 0–2 (multi-tenant core + offline POS/KOT + KDS) as the MVP that a single cloud kitchen can actually run on; this is your riskiest, highest-value muscle (offline-first + printing). Only then add inventory (Phase 4) and the aggregator connector (Phase 5), which is where most revenue relevance for a cloud kitchen lives.
2. **De-risk aggregators early in parallel:** start Zomato POS-vendor onboarding paperwork (NDA, form) the day you begin Phase 1, because certification + pilot is a multi-week external dependency. If you need orders flowing before certification completes, ship the **UrbanPiper/Dyno middleware adapter first** behind your platform-adapter interface, then swap to direct once approved.
3. **Treat the connector as a separate product** with its own uptime budget and on-call from day one — Zomato's 99.999% / <10-min SLA will get your integration disabled if you bolt it onto the main API.
4. **Don't build compliance or payments plumbing.** Integrate a GSP for IRN and Razorpay for money; store the returned artifacts. This is both faster and lower-liability.
5. **Benchmarks that change the plan:** if you stay single-brand/single-kitchen, you can defer Phase 7 multi-outlet complexity and RLS-heavy tenancy — but keep `tenant_id`/`outlet_id` in the schema from day one so you never have to retrofit. If order volume per outlet stays low, a hand-rolled delta-sync is fine; adopt a CRDT sync engine (ElectricSQL/PowerSync) only if concurrent multi-terminal editing conflicts become frequent. If you exceed a few hundred outlets or high webhook throughput, move the order-event bus to Kafka.

## Caveats
- **Integration counts are marketing figures.** Petpooja variously claims "80+", "150+" and "200+" integrations across its own pages and third-party listings; treat the specific partner list as representative, not exhaustive or exact. The ₹10,000/year entry price is a reseller-quoted figure and varies by region/add-ons.
- **Aggregator API specifics are partly gated.** Zomato's exact order/menu action sub-paths and request/response JSON live behind its API reference (accessible only to onboarded vendors); the lifecycle, status enums (`rider-assigned/pickedup/reject/timedout`), auto-reject window (~5 min), 99.999% uptime and the outlet endpoint paths (e.g. `/online-ordering/v1/restaurant_delivery_status/get|update`) are confirmed from public docs, but you must complete onboarding to get exact schemas and auth header/signature details. Swiggy publishes no public endpoint docs.
- **ONDC detail** is protocol-level; the RET11 domain and Beckn API set are stable, but seller-side food flows and settlement (RSF) evolve — verify against current ONDC reference apps at build time.
- **GST thresholds are moving.** The ₹5cr e-invoicing threshold and the 30-day/₹10cr reporting rule are current as of 2025; some sources anticipate a drop toward ₹2cr and a shorter reporting window — build the IRN integration to be threshold-configurable.
- **Some sourcing is secondary** (SoftwareSuggest/Techjockey/Slashdot/Scribd aggregator listings and a Petpooja-like spec on Scribd) and may lag the live product; treat exact feature availability per region as "confirm during onboarding," consistent with Petpooja's own "may or may not be available in your region" marketplace disclaimer.
